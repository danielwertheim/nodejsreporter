                    RSRP-337962
                    Do not move code out of lambda while applying "introduce variable" refactoring
                

            

            
I had the following code:

HighlightFileAndCheck(file, highlighting => HighlightingSettingsManager.Instance.GetSeverity(highlighting, file.ToSourceFile()) == severity, funcCheckErrors, solution);


Applied "introduce variable" refactoring to "file.ToSourceFile()", got this:

var psiSourceFile = file.ToSourceFile();
HighlightFileAndCheck(file, highlighting => HighlightingSettingsManager.Instance.GetSeverity(highlighting, psiSourceFile) == severity, funcCheckErrors, solution);


And because of that I lost read lock on file.ToSourceFile(), as lambda is executed under lock, but outer code is not.


If I had the following lambda instead:

HighlightFileAndCheck(file, highlighting =>
        {          
          return HighlightingSettingsManager.Instance.GetSeverity(highlighting, file.ToSourceFile()) == severity;
        }, funcCheckErrors, solution);


then refactoring worked fine:

HighlightFileAndCheck(file, highlighting =>
        {
          var psiSourceFile = file.ToSourceFile();
          return HighlightingSettingsManager.Instance.GetSeverity(highlighting, psiSourceFile) == severity;
        }, funcCheckErrors, solution);



So, refactoring should properly process lambdas without brackets by adding missing brackets and inserting new variable inside them, and not outside lambda. Otherwise, this leads to bugs.
